OmniRoute · open-sse · bugfix

Tool-name cloak map survives on the request-scoped ALS store

Stop the per-request _toolNameMap from being lost across the capture JSON.parse(JSON.stringify()) rebuild — carry it on the request-scoped AsyncLocalStorage instead of a fragile non-enumerable property. Kills the PascalCase my_* tool-name rejection on opencode ≥1.17.13.

01 The change — before & after

Before · fragile

Non-enumerable re-attach, per body shape

  • Cloak attaches _toolNameMap as a non-enumerable prop on the request body.
  • Capture body() rebuilds via JSON.parse(JSON.stringify()) → non-enumerable prop is dropped.
  • A re-attach hack copies it back from fallback._toolNameMap — patched 3× (#4091/#4153/#4181), one per new body shape.
  • opencode 1.17.13's new tools[] shape perturbs it → map empty at restore → HlLinearListProjects leaks → client rejects.
→
After · durable

Carry the map on the ALS capture store

  • New toolNameMap? field on the Capture type.
  • stashRequestToolNameMap() writes the active ALS store at cloak time (same request scope).
  • body() reads the map from the ALS store first; non-enumerable prop kept as fallback.
  • ALS is never serialized → cannot be dropped by any body-shape round-trip. Fragility eliminated.

02 Where the map lives across the request

Cloak runs inside the executor, which runs inside runWithCapture(). The ALS store is active at both map creation and re-attach — the same instance, same request scope.

Why durable, not just another patch. #4091/#4153/#4181 each re-attach the map at a specific serialization boundary — correct for the body shapes of their time, but a new opencode body shape can re-strand the map. The ALS store is never serialized and shares the executor's request lifetime, so it survives regardless of body shape. The non-enumerable re-attach stays as a backward-compatible fallback.
